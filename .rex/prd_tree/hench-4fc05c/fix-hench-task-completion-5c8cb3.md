description: "Fix two coupled hench completion-path bugs (GitHub #302): (A) task-completion metadata is written to `.rex/prd_tree` AFTER the run's code commit, so it is never committed and orphans in the working tree; (B) the hench config schema silently drops the top-level `fullTestCommand` key, so a configured full-suite command is ignored.\n\n## Context — architecture\n\nhench is the autonomous-agent tier (`packages/hench`). A per-task run lifecycle lives in `src/agent/lifecycle/shared.ts` (`finalizeRun` and helpers) and is driven by the CLI loop (`cli-loop.ts`) and API/Gemini loop (`loop.ts`). Config is loaded/validated through the Zod schema in `src/schema/validate.ts`, whose TypeScript counterpart is the `HenchConfig` interface in `src/schema/v1.ts`. Test-gate command resolution lives in `src/tools/test-command-resolver.ts`. All child-process/git calls go through `execStdout` from `src/process/exec.js`. This work is entirely intra-hench — no cross-package or gateway changes.\n\n## Bug A — completion metadata not committed by its own run\n\nPer-task order today: mark in_progress → agent edits files → test gate → the agent (CLI provider, `autoCommit` true) commits the CODE mid-run → hench marks the task completed and writes resolution metadata into `.rex/prd_tree`.\n\nRoot cause: in `finalizeRun` (`src/agent/lifecycle/shared.ts`), `updateCompletedTaskStatus` (~line 1714) writes the completed status + resolution fields to the folder tree BEFORE `performCommitPromptIfNeeded` (~line 1741). But when `autoCommit` is true, `performCommitPromptIfNeeded` is a no-op (see the comment ~line 1734: \"No-op when autoCommit is true (agent committed itself)\"). So the `.rex/prd_tree` metadata written for THIS task is never staged/committed — it is left dirty. The next `ndx work` pre-run commit gate (#279, `commitPreRunChanges` ~line 1085 which does `git add -A`) then trips on hench's own leftovers, and the just-finished task's completion metadata orphans.\n\nFix direction: after marking the task completed, commit the completion/resolution metadata in a small dedicated second commit that leaves a clean working tree — at minimum on the `autoCommit` path (where `performCommitPromptIfNeeded` does not run). Prefer a focused helper (e.g. `commitCompletionMetadata(projectDir, taskId)`) that stages the `.rex/prd_tree` changes (reuse the per-path `git add <prdPath>` pattern near line 1367 and the commit pattern near line 1471) and commits with a clear message plus the co-authored trailer (`buildCoAuthoredByTrailerLine`). Do NOT double-commit on the non-autoCommit path where `performCommitPromptIfNeeded` already stages the PRD paths alongside the code. Skip cleanly when the tree has no `.rex` changes or when not in a git repo.\n\n## Bug B — fullTestCommand silently dropped\n\n`HenchConfig` (`src/schema/v1.ts:163`) declares `fullTestCommand?: string`, and `test-command-resolver.ts` (line ~279) reads `options.config?.fullTestCommand`. But `HenchConfigSchema` in `src/schema/validate.ts` (the `z.object({...})` at lines ~52-77) does NOT list `fullTestCommand`. Zod's `z.object` strips unknown keys on parse, so `loadConfig` returns the config WITHOUT the field, and the resolver always falls back to auto-detect (`npm run test`) even when `.hench/config.json` sets `fullTestCommand`.\n\nFix direction: add `fullTestCommand: z.string().optional()` to `HenchConfigSchema` so the key survives validation. Verify the resolver then honors it.\n\n## Acceptance criteria\n\n1. A `fullTestCommand` set in `.hench/config.json` survives `validateConfig`/`loadConfig` (not stripped) and is used by the full-suite test gate instead of the auto-detected fallback.\n2. After a completed task run on the `autoCommit` path, the working tree has no uncommitted `.rex/prd_tree` changes — the completion/resolution metadata is committed by that run (verifiable via `git status --porcelain` being clean of `.rex` entries).\n3. The non-autoCommit commit-prompt path is unchanged (no double commit, existing behavior/tests still pass).\n4. A subsequent pre-run commit gate does not trip on the prior run's own completion metadata.\n5. New regression tests cover both fixes.\n\n## Testing guidance (TDD, red→green)\n\n- Unit: extend the config-validation tests (see `tests/unit/` schema/config tests) to assert `fullTestCommand` round-trips through `validateConfig`.\n- Integration: in a temp git repo (follow the harness in `tests/integration/rollback.test.ts` — `setupGitRepo`, `makeInitialCommit`), simulate a completed autoCommit run through `finalizeRun` and assert `git status --porcelain` reports no `.rex/prd_tree` leftovers afterward.\n- Windows note: git may restore/emit CRLF; normalize line endings in content comparisons (see the `.replace(/\\r\n/g, \"\n\")` pattern already used in `rollback.test.ts`).\n\n## Constraints\n\n- TDD: write failing tests first, then implement.\n- Do NOT regress the just-landed #303 scoped-rollback work in `shared.ts`/`review.ts`, or the existing commit-prompt flow.\n- Keep changes intra-hench; use `execStdout` for git, and existing helpers/patterns already in `shared.ts`.\n- `pnpm --filter @n-dx/hench typecheck` and the hench unit + rollback integration suites must stay green (modulo the ~38 pre-existing Windows CRLF/path failures tracked in #301)."
