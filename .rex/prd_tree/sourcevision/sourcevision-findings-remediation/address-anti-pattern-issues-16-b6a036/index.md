# Address anti-pattern issues (16 findings)

🔴 [completed]

## Summary

- Perfect cohesion (1.0) on a 28-directory, 149-file zone is a health-monitoring blind spot: mutual imports across all subsystems lock the cohesion score at 1.0 permanently, preventing automated split recommendations even as semantically unrelated concerns accumulate. Define and enforce a manual decomposition policy (e.g., >100 files OR >15 directories triggers a sub-zone audit) rather than relying on the cohesion metric alone.
- crash-detector.ts is directly importable from any file in the web package with no barrel enforcing single-entry-point access; future callers can bypass use-crash-recovery.ts and consume the detector directly, causing recovery state to diverge from detection state without any type-system or lint guard.
- The zone has two consumers at different architectural depths (web-application-core: 5 imports, web-dashboard: 1 import) but no barrel file defining a stable public API; any internal restructuring propagates to import sites in both zones simultaneously with no contract boundary to absorb the change
- hench-runs.ts is a full page/view that composes panel components, yet it lives in the dashboard-panel-components zone alongside the leaf components it consumes; this creates an intra-zone hierarchy inversion where a page-level file resides in a component-level zone, conflating two different abstraction levels and making it impossible to measure the zone's cohesion accurately without knowing which files are leaves versus composites
- polling-restart.ts has no unit test coverage; restart logic involves stateful transitions that integration and E2E tests are unlikely to exercise exhaustively, leaving a behavioral gap in the polling subsystem.
- use-polling.ts functions as a package-wide hub (4 importing zones) but is assigned to a satellite zone with cohesion 0.33; hub files in satellite zones misrepresent the zone's stability profile and make its coupling metric unreliable — the file's connectivity warrants placement in web-viewer or a dedicated polling-primitives layer.
- parse-gateway-exports.mjs audits gateway exports outside the test pipeline with no automated cross-check against domain-isolation.test.js verdicts; integrate the script into CI and diff its output against test assertions to surface drift at merge time rather than during manual audits.
- domain-isolation.test.js and parse-gateway-exports.mjs enforce overlapping gateway invariants via independent code paths with no shared source of truth; a gateway pattern update that modifies one without the other will pass CI undetected indefinitely.
- use-project-status.ts hook likely has zero dedicated test coverage: the zone's only test file matches the component name, not the hook name, creating an untested stateful module in a zone that already has the minimum viable file count.
- No barrel file exists for the zone, so web-viewer must maintain 3+ direct import paths to individual page module files; any internal rename or split requires coordinated multi-site updates in web-viewer with no single entry point to update.
- Three page components that render sourcevision JSON output have zero test coverage and no schema contract; a breaking change to sourcevision's output format will corrupt all three views silently, with no unit, integration, or snapshot test to detect the regression.
- usage-cleanup-scheduler.ts imports loadPRDSync from prd-io.ts (line 37) directly rather than receiving it through the same dependency-injection channel used for collectAllIds; this undocumented direct coupling partially undermines the DI inversion and means the zone has a second, uncontrolled dependency on web-application-core internals that will not be caught by the existing DI pattern documentation
- web-application-core's 0.92 cohesion metric aggregates both server-layer and viewer-layer files into a single score, hiding the bidirectional imports between these layers (web-viewer→web-server: 4, web-server→web-viewer: 1) that would be flagged as upward-dependency violations under CLAUDE.md's web-internal zone stack if the zone were correctly split
- web-dashboard reaches directly into the polling infrastructure zone (3 imports) bypassing web-application-core as the integration hub; in the stated hub-and-spoke model all cross-zone dependencies from satellite zones should route through web-application-core, making these 3 lateral edges undocumented topology violations
- boundary-check.test.ts flags the web-viewer → web-server import question as an open concern but does not assert against it; the test responsible for enforcing web-internal layering has an unverified gap for the single highest-risk cross-zone edge in the package.
- God function: cmdAnalyze in packages/sourcevision/src/cli/commands/analyze.ts calls 32 unique functions — consider decomposing into smaller, focused functions
