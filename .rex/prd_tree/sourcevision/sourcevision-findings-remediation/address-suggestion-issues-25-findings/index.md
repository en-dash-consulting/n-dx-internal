## Summary

- Add a build-file exclusion list to sourcevision configuration: packages/web/build.js, packages/web/dev.js, and any future *.config.js or build-tooling files should be excluded from zone membership analysis. These files share directory proximity with runtime components but have no runtime import relationships with them — their inclusion corrupts cohesion and coupling metrics for any zone they land in.
- Add a hooks/index.ts barrel to packages/web/src/viewer/hooks/ and a boundary-check.test.ts assertion requiring consumers to import through the barrel. The hooks directory is the largest leaf-import surface in the web viewer (29 files) with no enforcement — analogous to the pre-barrel state of crash/ and shared/ before their boundary assertions were added.
- CLAUDE.md describes crash as having 'bidirectional coupling with web-viewer' but current imports show only web-viewer → crash (viewer imports crash's recovery utilities) with crash importing web-shared directly (documented bypass) rather than web-viewer — update the governance table to accurately describe the actual dependency direction to prevent engineers from applying the wrong mitigation strategy.
- Consolidate the 'Required Tests' table into TESTING.md as the single source of truth. Replace the corresponding CLAUDE.md table entry with a reference link to TESTING.md. Add an architecture-policy test assertion that scans required test file paths for the 'REQUIRED TEST' annotation string, making the required-test contract machine-verifiable in CI rather than documentation-only.
- Create a monorepo-level 'injection seam registry' alongside the gateway module table in CLAUDE.md. Seams that use callback injection (start.ts → register-scheduler.ts) rather than gateway imports are invisible to static analysis — listing them explicitly with their injected interface types prevents future contributors from inadvertently replacing injection with direct imports to 'simplify' the code.
- Extend build-output-contract.test.ts to cover server bundle artifacts (dist/server/ or equivalent). The current contract is browser-only (viewer + landing), leaving server build regressions undetected in CI until a deployment failure.
- Neither the rex package nor the hench package has an intra-package one-way dependency direction test despite having 4 active bidirectional cycles between them (rex: 3, hench: 1) — adding a single assertion per package that no file in src/ imports from a higher-tier directory unless it is itself in that tier would provide the same CI regression protection that web's boundary-check.test.ts provides for the web package.
- The current zone analysis data contains at least one confirmed phantom edge (web-viewer → web-server: 39) that has been driving architectural governance discussions for multiple analysis passes. Before the next governance pass, re-run sourcevision analysis to produce a fresh zone map. All architectural priorities derived from stale cross-zone edge counts should be treated as provisional until confirmed by fresh analysis.

The foundation tier is the most critical boundary in the four-tier hierarchy — closing this gap requires either removing the wildcard export (breaking change) or adding zone-level entry point assertions to sourcevision configuration (non-breaking).

Zone detection should be re-run with the full rex CLI directory in scope, or a catch-all 'rex-cli-commands' zone should be explicitly defined to surface the ungoverned CLI surface.

The confirmed 4 dynamic import sites in analyze.ts→chunked-review are invisible to zone coupling metrics. A supplementary CI check scanning for cross-zone dynamic imports would close this systematic enforcement blind spot before it grows.
- Audit the 12 entry points to determine which are prd-tree exports (which would move with reclassification) vs. genuine platform-level exports. If fewer than 4 entry points remain after prd-tree moves, the residual zone is a dissolution candidate rather than a consolidation target.
- Audit all 39 web-viewer → web-server imports to verify each passes through rex-gateway.ts or domain-gateway.ts. A single direct leaf import at this seam would be the highest-risk gateway violation in the monorepo given the 39-import volume. Add a CI assertion that no file outside src/viewer/rex-gateway.ts or src/viewer/domain-gateway.ts imports directly from src/server/.
- The web-viewer → web-server: 39 imports cross-zone edge is confirmed zero in the current codebase (web-viewer is a browser SPA that cannot import server modules at runtime). All governance priorities built around this being the highest-volume coupling edge are based on stale zone analysis data.
