# Address suggestion issues (20 findings)

🔴 [completed]

## Summary

- Add a non-importable file extension exclusion list to sourcevision zone detection (PNG, JPG, SVG, GIF, HTML, CSS, MD, JSON, YAML, lock files). Binary assets and markup files cannot form import edges, so including them in zone detection produces noise zones (logo, web-landing) with false-positive 'at-risk' labels and distorted cohesion scores of 0. This single change would eliminate at least 3 noise zones and remove misleading risk labels from the zone map.
- Add subdirectory grouping to packages/rex/src/cli/commands/ to make the three-zone split visible in the file tree. Create commands/chunked-review/ and commands/fix/ subdirectories for the satellite zone files, leaving the remaining 27+ core-zone files in the flat commands/ root. This mirrors the convention used elsewhere in the monorepo and gives the community detection algorithm a directory signal to produce stable zone assignments.
- CLAUDE.md's web-shared governance section states it is 'the only production zone meeting both fragility thresholds simultaneously' — this is factually incorrect. rex-cli (0.25/0.75) and fix (0.25/0.75) also meet both thresholds. Update the documentation to list all three dual-fragility zones and apply consistent governance to each.
- Establish a monorepo-wide zone governance policy that applies the dual-threshold fragility test (cohesion < 0.5 AND coupling > 0.5) uniformly across all packages — not just web-shared. Both rex satellite zones currently breach both thresholds but have no governance response. A single governance rule in CLAUDE.md covering all packages prevents the current situation where identical metric signatures receive different treatment depending on which package they appear in.
- The implicit web-package load order (web-shared → viewer-message-pipeline → web-dashboard-platform → web-server) has no mechanical enforcement. Add a zone-cycle assertion in boundary-check.test.ts or architecture-policy.test.js that validates no new import cycle is introduced across these four zones — the existing zone-cycle detection covers package-level cycles but not intra-web-package load-order violations, leaving the web package's internal layering unguarded.
- boundary-check.test.ts uses regex-based import path extraction that is blind to dynamic imports, making it structurally unable to enforce policies against the dynamic import patterns already identified in rex (analyze.ts) and any future dynamic imports in web. Supplement boundary-check.test.ts with a separate dynamic-import-audit test (or a grep-based CI check) that enumerates all 'await import()' call sites across package source directories and asserts that none cross zone boundaries without documentation.
- hench-agent (160 files, 31 directories) has no documented intra-zone governance despite being the second-largest zone in the monorepo. Establish a governance document for hench parallel to CLAUDE.md's web-shared section: identify sub-zone boundaries (agent/, prd/, brief/), document index.ts barrel requirements, and add boundary assertions to hench's test suite before the zone reaches web-viewer's scale.
- Add a ZONE_BOUNDARY marker comment or README to packages/rex/src/cli/commands/ explaining which files belong to the satellite zones (chunked-review, prd-fix-command) vs the core zone. A flat directory with three zone memberships and no sub-directories creates invisible boundaries that only sourcevision can detect.
- The production→test import from web-dashboard-platform into web-unit combined with 7 reverse imports from web-unit back into web-viewer forms a closed production↔test cycle, not a simple one-directional violation. Treat remediation as requiring both edges to be resolved simultaneously — eliminating only the production→test direction will leave a unidirectional cycle that still couples test zone stability to production API changes.
- Establish a rex-satellite governance policy in CLAUDE.md analogous to the web-shared policy: both satellite zones (chunked-review, prd-fix-command) have cohesion 0.25 and coupling 0.75, meeting the same dual-threshold condition that triggered web-shared governance. Without a policy, these zones will continue accepting new files with no fragility guardrail.
- crash (cohesion 0.5, bidirectional web-viewer coupling) is one poorly-placed addition away from dual-fragility. Apply the two-consumer rule proactively to new crash zone additions before cohesion degrades further.
- fix has cohesion 0.25 and coupling 0.75 — both exceed fragility thresholds, yet CLAUDE.md's governance section does not list it as a dual-fragility zone. Apply the same active governance policy applied to web-shared (addition review, consumer count checks) to fix immediately.
- Relocate packages/rex/src/core/fix.ts to rex-prd-engine by auditing whether its imports are exclusively to other rex-prd-engine files. If confirmed, the file was misassigned to prd-fix-command by import-proximity clustering. Moving it removes the zone's only non-CLI file and makes the zone name accurate.
- rex-cli has cohesion 0.25 and coupling 0.75 — both exceed fragility thresholds, matching the profile CLAUDE.md reserves exclusively for web-shared. Update CLAUDE.md to acknowledge rex-cli as a dual-fragility zone and apply equivalent active governance.
- shared-types.ts (permanent, cycle-breaking, wide blast radius) and the scheduler services (transient, event-driven, narrow blast radius) have incompatible stability contracts but share a zone. Consider moving shared-types.ts to src/shared/ to correctly reflect its zone-neutral role and decouple its governance from the transient service lifecycle — a change to shared-types.ts should trigger broader review than a change to register-scheduler.ts, but the current zone co-location makes them indistinguishable.
- The 'web' zone's bidirectional coupling with web-viewer (16 in + 6 out) combined with its 2 imports from web-server creates a hidden secondary coupling path web-server→web→web-viewer that inflates the real viewer-server dependency beyond the direct 37-edge count. Audit the 6 web-viewer→web edges and 2 web→web-server edges to determine whether they represent the same type contracts as the direct viewer→server edges, making the secondary path redundant and removable.
- The web↔web-viewer cycle (2.7:1 ratio) is the most tightly mutually dependent cycle in the monorepo. Unlike lopsided cycles resolvable by elevating a few reverse imports through a shared interface, this cycle likely requires a structural decision: fully absorb 'web' into web-viewer, split its responsibilities into web-server, or formalize it as an explicit composition shell with a documented API contract.
- boundary-check.test.ts uses regex-based import extraction, which is blind to dynamic imports (await import()), template-literal paths, and multiline import statements. Any new enforcement assertion (including the proposed two-consumer rule check) will inherit these blind spots. Document this limitation in a comment at the top of boundary-check.test.ts and consider whether the two critical dynamic import paths in analyze.ts and cli/index.ts need a separate enforcement mechanism.
- web-landing's summary.json reports fileCount: 2 but the files array contains 3 entries (index.html, landing.css, landing.ts) — a data integrity mismatch. Verify the zone serialization path handles HTML/CSS file counting correctly and does not silently drop non-TypeScript files from the count.
- shared/index.ts is the only cohesion mechanism in this zone — it is a barrel exporting 3 constants and 1 type from two semantically unrelated files. No automated check enforces that consumers import through index.ts rather than directly from data-files.ts or view-id.ts. If direct imports proliferate, zone cohesion becomes unmeasurable and the two-consumer rule cannot be applied per-module. boundary-check.test.ts should assert that web-shared consumers import through the barrel index rather than from leaf files directly.
